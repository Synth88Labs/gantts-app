关键路径法：算法、公式与算例

关键路径是项目中相互依赖的任务里最长的一条链。它的长度就是项目最短可能工期——这条链上每延误一天，完工就晚一天。

基本想法

并非每项任务对工期的影响都一样。有些有余量：可以推迟而不影响完工。有些一点余量都没有——这些构成关键路径。

这个方法叫关键路径法（Critical Path Method，CPM），源自 1950 年代末。它回答两个问题：项目最少要多久，以及哪些任务绝对不能拖。

正推：最早时间

从项目开始向前推算。对每项任务：最早开始时间取所有前置任务最早完成时间中的最大值；最早完成时间等于最早开始加工期。

正推结束时，就得到了项目的最短可能工期。

逆推：最晚时间

再从项目结束往回推。最晚完成时间取所有后续任务最晚开始时间中的最小值；最晚开始时间等于最晚完成减工期。

浮动时间与关键路径

某任务的总浮动时间 = 最晚开始 − 最早开始。浮动时间为零的任务构成关键路径。关键路径至少存在一条，在依赖密集的计划中往往有多条并行。

一个四任务算例

A 工期 3 天；B 工期 5 天，紧接 A；C 工期 2 天，同样紧接 A；D 工期 4 天，等 B 和 C 都完成。

正推：A 从第 0 天到第 3 天。B 从 3 到 8，C 从 3 到 5。D 要等两者，因此从 8 开始，到 12 结束。项目工期：12 天。

逆推：D 必须在第 8 天开始，所以 B 必须在 8 完成——最晚开始为 3，浮动时间 0。C 也必须在 8 完成，但只需 2 天：最晚开始为 6，而实际最早开始是 3，浮动时间 3 天。

关键路径是 A → B → D。C 可以推迟三天而不影响完工。

这在实际中意味着什么

它告诉你精力花在哪里有效。给 C 加人对工期毫无帮助；在 B 上省下一天，项目就早一天完工。

它还提示一个陷阱：如果 C 推迟四天，C 自己就变成关键任务。关键路径不是计划的固有属性，而是某一时刻的快照——只要有变动，它就会转移。

关键路径在项目过程中会变。一项有三天浮动时间的任务若延误四天，它自己就成了关键任务。因此关键路径不是开工时算一次，而是每次更新都要重算。

常见问题

什么是关键路径？

相互依赖的任务中最长的一条链。它的长度等于项目最短工期，链上任务没有浮动时间。

关键路径怎么计算？

先正推求最早时间，再逆推求最晚时间。两者相同、即浮动时间为零的任务，构成关键路径。

什么是总浮动时间？

一项任务在不推迟项目完工的前提下可以推迟的时间：最晚开始减最早开始。

关键路径可能有多条吗？

可能。在依赖密集的计划中，多条并行路径长度相同，于是全部成为关键路径，风险随之显著上升。
