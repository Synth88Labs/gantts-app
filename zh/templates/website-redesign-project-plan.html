模板包含什么

改版很少败在设计上。它们败在两条没人配足人手的不起眼轨道上——把内容搬过去，和把 URL 映射好——所以模板给这两条各留了独立的阶段和独立的依赖关系：

  • 需求调研 — 干系人访谈、流量与搜索数据复盘、竞品与用户研究，以及达成一致的需求和成功指标。里程碑：需求确认。
  • 信息架构与内容盘点 — 完整 URL 清单与全站抓取、对每一个页面做保留/合并/下线的决策、站点地图与导航设计和关键模板的线框图。里程碑：信息架构确认。
  • 设计 — 视觉方向、设计系统与组件、关键页面设计，以及响应式与无障碍评审。里程碑：设计确认。
  • 开发 — 环境与 CMS 搭建、模板与组件开发、接口与表单，以及数据统计埋点。里程碑：功能开发完成。
  • 内容迁移 — 迁移对照表、内容改写与优化、图片视频迁移和真正导入 CMS。里程碑：旧站内容冻结。
  • SEO 与跳转映射 — 为每一个变化的页面建立旧 URL 到新 URL 的映射、跳转规则搭建与测试、元数据与结构化数据、canonical 标签和 XML 站点地图。里程碑：跳转映射确认。
  • 测试 — 跨浏览器与多设备测试、无障碍检查、Core Web Vitals、在预发布环境完整抓取验证跳转和干系人验收测试。里程碑：测试通过。
  • 上线与监测 — 上线评审、DNS 切换、上线后跳转复核、站点地图提交，以及三十天的抓取错误与排名监测。
跳转映射就是这个项目本身。每一个变了地址却没有 301 的 URL，都是你花钱买来又扔掉的流量，而且损失要几周之后、排名已经掉了才显现。映射表要从旧站的完整抓取生成——不是从新站点地图生成——并且要在上线之前就在预发布环境测通，不是上线之后。

如何调整

  1. 第一周就开始做 URL 清单；旧站的抓取结果是内容盘点、迁移对照表和跳转映射三者共同的输入。
  2. 如果要迁移的是上千个页面，按内容类型或栏目加行——一根"迁移内容"的横条永远藏着一个月。
  3. 内容改写的横条要拉到诚实的长度；它通常是整个项目里最被低估的一项，而且需要的是写手，不是开发。
  4. 在旧站保留一个内容冻结的里程碑，别让迁移去追一个还在动的目标。
  5. 把预发布环境的跳转抓取设成上线评审的硬前置任务，而不是测试阶段的一项锦上添花。
  6. 上线后监测的横条至少拉到三十天——搜索引擎重新抓取并重新处理一个改版站点要花好几周。

排期建议

  • 先抓取旧站，别的都往后放。没盘点过的东西没法盘、没法迁、也没法做跳转，而大多数组织线上的 URL 数量都远比自己以为的多。
  • 不要在周五上线。跳转和收录问题几个小时内就会冒出来，需要有人在岗处理。
  • 用真实映射表在预发布环境测跳转。抽查十个 URL 什么也说明不了；要把整份旧 URL 清单对着预发布环境跑一遍，逐个核对响应码。
  • 保留旧的统计口径并行一段时间。没有可比的基线，你分不清上线后流量下滑是季节性的还是自己搞出来的。
  • 内容要在模板做完之前就开始写。内容等设计，是上线日期连着推迟两次最常见的原因。
  • 在设计确认时设定基准。之前范围还在动；之后指导小组该看的就是偏差了。

常见问题

网站改版要多久？

中等规模站点通常 4 到 9 个月，主要取决于内容量，而不是设计或开发。这份模板用的是大约九个月的计划，压缩要靠砍范围，不能靠砍迁移时间。

跳转映射是什么，为什么重要？

它是一张逐条列出每个旧地址对应新地址的表，通过 301 跳转实现。没有它，每一个变了地址的 URL 都会丢掉积累的搜索权重和指向它的外链。正因如此它在这里单独成了一个阶段。

内容迁移什么时候开始？

内容盘点做完、对照表有了就开始——远早于开发结束。内容是最长的一根杆子，等模板做好再开始会多出好几个月。

上线之后应该监测什么？

抓取错误与 404、收录情况、重点关键词的排名，以及对照上线前基线的自然流量，至少监测三十天。这四项在图上都是真实任务。
